## Re-verification (2026-03-17) — Run 13 — 204 tests

### Context
Test suite grew from 200 → 204 tests after additional test classes were added.
The original SseEmitter fix was insufficient for the full suite. Multiple additional
fixes were required before the suite passed cleanly.

### Fixes applied (cumulative, Run 1 → 13)

**Fix 1 — TestRedisConfig.java** (`springSessionContainerPatcher`):
- Root cause: `springSessionRedisMessageListenerContainer` did `Thread.sleep(5000)` on
  reconnect when subscription executor was not a `ScheduledExecutorService`.
- Fix: `setSubscriptionExecutor(new ScheduledThreadPoolExecutor(2))` in BeanPostProcessor.

**Fix 2 — RetroSession.java** (`advancePhase()`):
- Root cause: `RetroPhase.values()[getPhase().ordinal() + 1]` → after `CLOSE_RETRO`
  (ordinal 6), ordinal+1=7=`PAUSED` instead of `COMPLETED`.
- Fix: `getPhase().next()` used instead.

**Fix 3 — SseBrowserTest.java**:
- Root cause: Skip loop began before participant page finished rendering first step.
- Fix: `waitForElement(participantPage, "[data-step-index]", DEFAULT_TIMEOUT_MS)` added
  before skip loop.

**Fix 4 — MultiUserRetroBrowserRegressionTest.java**:
- Root cause: Bob/Carol `isVisible()` assertions fired without waiting for network
  round-trip + React re-render.
- Fix: `waitForElement(bobPage, ...)` / `waitForElement(carolPage, ...)` added before
  each `assertTrue`.

**Fix 5 — MultiUserRetroBrowserRegressionTest.java**:
- Root cause: `waitForResponse` with 15s timeout fired before Tomcat thread pool drained
  ~30 active SSE connections (saturated at COMPLETED phase advance).
- Fix: Replaced `waitForResponse` block with direct `.click()` +
  `waitForFunction("data-phase=COMPLETED")` at `SSE_PROPAGATION_TIMEOUT_MS * 2 = 30000ms`.

**Fix 6 — src/test/resources/application.yaml**:
- Root cause: After all 204 tests complete and Surefire calls `System.exit(0)`, non-daemon
  Tomcat async threads holding open SSE `DeferredResult` connections keep JVM alive for ~31s.
  Lettuce/Netty EventLoopGroup also non-daemon, adding more hold time.
- Fix: `server.shutdown=graceful` + `spring.lifecycle.timeout-per-shutdown-phase=5s`.
  When Spring's shutdown hook fires, graceful shutdown drains in-flight requests (max 5s),
  then force-closes all async threads and properly shuts down Lettuce/Netty EventLoopGroup.
