Task 13: Core Facilitation Split Evidence

1. Controller Decomposition
The monolithic RetroApiController was split into three focused controllers:
- SessionApiController (facilitation.session): Handles lobby creation and flow progression (start, next, timer).
- ParticipantApiController (facilitation.participant): Handles participant join/leave and participant list retrieval.
- ResponseApiController (facilitation.response): Handles note submission (column/rating), voting, reveal, and update.

2. Test Decomposition
Correspondingly, the unit tests were split:
- SessionApiControllerTest: 5 tests covering session lifecycle.
- ParticipantApiControllerTest: 3 tests covering join/leave flows.
- ResponseApiControllerTest: 4 tests covering data submission, reveal, and update.

3. Package Structure
Production code now resides in:
- direct.reflect.facilitator.facilitation.session
- direct.reflect.facilitator.facilitation.participant
- direct.reflect.facilitator.facilitation.response

DTOs were moved to respective sub-packages where specific to a capability:
- facilitation.session.dto (6 files: AssistantMessageDto, AssistantStateDto, CreateRetroRequest, CreateRetroResponse, NextStepResult, TimerStateDto)
- facilitation.participant.dto (5 files: JoinRetroRequest, JoinRetroResponse, LeaveActiveSessionsResult, ParticipantDto, SessionInfo)

The following DTOs remain intentionally in direct.reflect.facilitator.facilitation.dto as they are shared or multi-purpose:
- ColumnResponseDto, ComponentResponseDto, RatingResponseDto, RetroStateDto, RevealResult, StageProgressDto, StepSummaryDto, SubmitResponseResult, SyncVersionedResponse, UpdateResponseResult, VoteResult.
